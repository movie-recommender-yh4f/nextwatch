`/api/gemini/recommend` generates fresh recommendations from watched history and My List context. Use `/api/recommend` when you want cache-aware behavior.

## Auth

- Requires `Authorization: Bearer <supabase_access_token>`
- User is resolved server-side from the token
- Requests without a valid token return `401 Unauthorized`

## How It Works

1. Authenticates the user via Supabase JWT token
2. Fetches the user's watched movies from `user_watched_movies` and My List IDs from `user_my_list`
3. Hydrates both sets from the Supabase `movies` table so Gemini sees titles and years
4. Merges optional request-body movies into the watched prompt input
5. Sends watched history + My List context to Gemini with structured JSON output schema
6. Resolves each recommendation to a TMDB ID by searching the Supabase `movies` corpus first and falling back to TMDB search when needed
7. Returns up to 20 recommendations

## Endpoint

### `POST /api/gemini/recommend`

Generates up to 20 movie recommendations based on watched movies, with My List treated as low-priority input.

The request body is optional. If omitted, Supabase watched history plus Supabase My List are used. `id` refers to the movie's TMDB ID.

::code-group
```json [Request (optional)]
{
  "movies": [
    { "id": 550, "title": "Fight Club", "year": 1999 },
    { "id": 278, "title": "The Shawshank Redemption", "year": 1994 }
  ]
}
```

```json [Response]
{
  "recommendations": [157336, 155, 27205]
}
```
::

In development, the route returns debug objects instead of plain IDs:

```json
{
  "recommendations": [
    { "tmdbId": 157336, "originalName": "Interstellar", "year": 2014 }
  ]
}
```

#### Behavior

- Fetches user's Supabase watch history
- Fetches user's Supabase My List entries
- Merges with body movies (if provided), deduped by TMDB ID
- Returns `400` if no watched movies exist
- Sends watched + merged list to Gemini with structured output schema (name + year), while treating My List titles as low-priority candidates
- At most one recommendation can come from My List; unseen titles outside My List are preferred
- Validates response and returns up to 20 recommendations with resolved TMDB IDs

## Rate Limiting

Requests are limited to **20 per day** per authenticated user (sliding window).

The quota is checked before calling Gemini. A token is only consumed when at least one model returns a non-503 response. If all models in the fallback chain return `503 Service Unavailable`, the request does not count against the user's daily limit.

The following headers are included in every response:

| Header                  | Description                                |
| ----------------------- | ------------------------------------------ |
| `X-RateLimit-Limit`     | Maximum requests allowed per day           |
| `X-RateLimit-Remaining` | Requests remaining in the current window   |
| `X-RateLimit-Reset`     | Unix timestamp (ms) when the window resets |

Exceeding the limit returns `429 Too Many Requests`.

::callout{type="warning"}
When the daily limit is reached, the client should stop retrying and ask the user to try again later.
::

## Model Fallback

When the primary model (`gemini-flash-lite-latest`) returns `503`, the request is automatically retried with `gemini-2.5-flash-lite`, then `gemini-2.0-flash`. If all three models return `503`, the endpoint returns `503 Service Unavailable` without consuming a rate-limit token.

## Error Codes

- `400` no watched movies found
- `401` missing or invalid bearer token
- `429` daily recommendation limit reached
- `500` Supabase query failure or Gemini API key not configured
- `502` Gemini returned invalid JSON or response did not match schema
- `503` all Gemini models temporarily unavailable (token not consumed)

## Environment

Requires `NUXT_GEMINI_API_KEY` to be set in `.env`. If missing, the endpoint returns `500`.
