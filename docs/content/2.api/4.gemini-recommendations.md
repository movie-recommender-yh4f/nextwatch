Gemini generation happens inside `GET /api/recommend`, which handles auth, cache reuse, regeneration, My List context, and fallback behavior in one place.

## How It Works

1. Authenticates the user via a Supabase JWT token
2. Fetches the user's watched movies from `user_watched_movies`
3. Fetches the user's My List IDs from `user_my_list`
4. Builds the Gemini prompt from watched movies plus My List context
5. Calls Gemini with structured output expectations
6. Resolves recommendation names back to TMDB IDs
7. Returns valid IDs and caches them in Supabase

## Models

The generation helper uses this fallback order:

1. `gemini-flash-lite-latest`
2. `gemini-2.5-flash-lite`
3. `gemini-2.0-flash`

If all models fail with `503`, the request returns `503 Service Unavailable`.

## Rate Limiting

Recommendation generation is limited to **20 requests per day** per authenticated user.

The following headers are included on generation requests:

| Header | Description |
| ----------------------- | ------------------------------------------ |
| `X-RateLimit-Limit` | Maximum requests allowed per day |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp (ms) when the window resets |

## Failure Behavior

When regeneration fails and an older cached set exists, `/api/recommend` can return:

- `regenerationError`
- `staleRecommendations`
- `unmatchedRecommendations` in development

## Environment

Requires `NUXT_GEMINI_API_KEY` to be set in the app environment. If it is missing, recommendation generation returns an error from `/api/recommend`.
