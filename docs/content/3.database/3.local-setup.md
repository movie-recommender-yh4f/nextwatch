The LibSQL database is a plain SQLite file at `database/local.db`. It does **not** exist in the repository - you create and populate it yourself.

## Prerequisites

- Node.js v18+
- The app dev server running (`npm run dev` inside `app/`)
- `ADMIN_API_TOKEN` set in `app/.env` (or `app/.env.local`)
- `NUXT_LIBSQL_URL=file:../database/local.db` set in `app/.env` (or `app/.env.local`)

::steps
### Apply the Schema

Run the schema SQL against the local database file once to create all tables, triggers, and indexes.

```bash
sqlite3 database/local.db < database/schema.sql
```

After this step, `database/local.db` exists with the correct structure but all tables are empty.

### Populate the Movie Index

Trigger the TMDB import to download and insert all movies. This downloads yesterday's TMDB export file, filters out adult and low-popularity entries, and writes the remaining rows into `movies_index`.

With the dev server running:

  :::code-group
  ```bash [bash]
  curl -X POST http://localhost:3000/api/admin/tmdb-import \
    -H "x-admin-token: $ADMIN_API_TOKEN"
  ```

  ```powershell [PowerShell]
  Invoke-RestMethod -Method Post `
    -Uri "http://localhost:3000/api/admin/tmdb-import" `
    -Headers @{ "x-admin-token" = $env:ADMIN_API_TOKEN }
  ```
  :::

A successful response looks like:

```json
{
  "imported": 312847,
  "skipped": 14,
  "adultExcluded": 4201,
  "lowPopularityExcluded": 188932,
  "durationSeconds": "360"
}
```

Once this completes, `movies_index` and `movies_fts` are populated and recommendation title matching has local data to resolve TMDB IDs.

The default run skips TMDB rows whose IDs are already at or below the current maximum `tmdb_id` in `movies_index`, writes new rows in batches of 1500, and rebuilds the FTS index after the import finishes.

If you need to refresh existing titles as well, call the same endpoint with `fullRefreshQuery=true`:

  :::code-group
  ```bash [bash]
  curl -X POST "http://localhost:3000/api/admin/tmdb-import?fullRefreshQuery=true" \
    -H "x-admin-token: $ADMIN_API_TOKEN"
  ```

  ```powershell [PowerShell]
  Invoke-RestMethod -Method Post `
    -Uri "http://localhost:3000/api/admin/tmdb-import?fullRefreshQuery=true" `
    -Headers @{ "x-admin-token" = $env:ADMIN_API_TOKEN }
  ```
  :::

### Verify

Check that rows exist in the index:

```bash
sqlite3 database/local.db "SELECT COUNT(*) FROM movies_index;"
```

And that the FTS index is working:

```bash
sqlite3 database/local.db \
"SELECT tmdb_id, original_title FROM movies_fts WHERE movies_fts MATCH 'inception' LIMIT 5;"
```
::

## Switching to Turso (Remote LibSQL)

To use a hosted Turso database instead of the local file, update two env vars - no code changes are needed:

```bash
NUXT_LIBSQL_URL=libsql://your-db.turso.io
NUXT_LIBSQL_AUTH_TOKEN=your_turso_auth_token
```

Then apply the schema via the Turso CLI:

```bash
turso db shell your-db < database/schema.sql
```

And trigger the import the same way as [Step 2](/database/local-setup#populate-the-movie-index).

## Troubleshooting

### `no such table: movies_index`

The schema has not been applied. Run [Step 1](/database/local-setup#apply-the-schema).

### Import returns `502`

The TMDB export URL for yesterday may not have been available yet, or the import may have hit a stream, network, or database failure. SQLite batch writes retry up to 5 times on `SQLITE_BUSY` with a short linear backoff, but a `502` means the overall import still failed.

### `NUXT_LIBSQL_URL is not set`

Add `NUXT_LIBSQL_URL=file:../database/local.db` to `app/.env` (or `app/.env.local`) and restart the dev server.

### `database/local.db` is gitignored

This is intentional - the database file is large and regenerable. The `.gitignore` in `database/` excludes `*.db`.
